Building a Client-Side
PDF Engine

How ihatepdf.cv processes 150MB files entirely in your browser. WebAssembly, memory management, device adaptation, offline-first architecture — and why privacy-as-architecture is fundamentally different from privacy-as-policy.

On this page

    The problem nobody talks about

    If you've ever used an online PDF tool, you've accepted a quiet trade-off: your document — the contract, the CV, the medical record — traveled over the internet to a stranger's server, was processed by software you can't inspect, and was stored somewhere under a privacy policy you never read.

    Processing PDFs is computationally expensive. A 50MB PDF with embedded images needs 200–400MB of RAM. Professional PDF libraries like Ghostscript are written in C — not JavaScript. JavaScript was never designed for heavy binary file manipulation. This is why every major "free" PDF tool — iLovePDF, Smallpdf, Adobe Acrobat online — processes files on servers. It's not laziness. It's a genuine technical constraint.

    WebAssembly changed that.

    What WebAssembly actually does

    WebAssembly (WASM) is a binary instruction format that lets code written in C, C++, and Rust run inside a browser at near-native speed. The same Ghostscript that Adobe and professional print shops use — compiled to WebAssembly — now runs inside a browser tab.

    ihatepdf.cv uses three core WebAssembly-powered libraries:

    • pdf-lib — PDF manipulation: merging, splitting, watermarks, encryption, metadata. Works at the PDF structure level — copies pages between documents without re-rendering them. That's why merge operations are nearly instant regardless of file size.
    • PDF.js — Mozilla's PDF rendering engine, same one Firefox uses natively. Converts pages to Canvas elements for pixel-perfect previews and high-DPI image export. Worker architecture offloads heavy parsing to a separate thread, keeping the UI responsive.
    • Ghostscript (WASM) — compression. The professional-grade engine behind the 60–70% file size reductions. Five simultaneous optimizations: image downsampling via bicubic interpolation, JPEG recompression, font subsetting (up to 90% reduction), metadata stripping, stream recompression. Text and vectors completely unaffected.

    The memory problem

    A 50MB PDF doesn't require 50MB of RAM to process — it requires 200–400MB. The expansion factor is real. When PDF.js renders a page to a Canvas element, it allocates width × height × 4 bytes (RGBA pixels). A Letter-size page at 300 DPI is 2550 × 3300 pixels — roughly 32MB per page. At 600 DPI: 134MB per page.

    Problem 1 — Canvas memory isn't just RAM

    Each canvas element allocates GPU texture memory in addition to system RAM. A 4000×6000 canvas uses 96MB RAM plus 96MB VRAM. On mobile with shared memory, that's 192MB gone per page. Without explicit cleanup: 100 pages × 96MB = 9.6GB accumulated.

    // This is not enough — GC may not run for seconds
canvas = null;

// This is what actually releases GPU memory immediately
canvas.width = 0;
canvas.height = 0;
canvas = null;

    Setting dimensions to zero tells the browser to release the GPU resource immediately, not wait for garbage collection.

    Problem 2 — JavaScript's GC is lazy

    The GC won't free memory until pressure is high. Processing 100+ pages simultaneously would crash any browser. The solution is deliberate batching with explicit GC pauses:

    const processBatches = async (allPages, batchSize) => {
  for (let b = 0; b < batches.length; b++) {
    for (const page of batches[b]) {
      const canvas = await renderPage(page);
      await downloadImage(canvas);
      canvas.width = canvas.height = 0; // free immediately
    }
    // 2-second pause — Chrome GC triggers after ~1–1.5s idle
    await new Promise(r => setTimeout(r, 2000));
    if (window.gc) window.gc(); // hint, not guarantee
  }
};

    The 2-second delay is deliberate. Chrome's GC heuristics trigger after ~1–1.5 seconds of idle. Without this pause, memory accumulates until the tab crashes.

    Device-adaptive processing

    The same codebase runs on a 2GB RAM phone and a 32GB workstation. The approach: scale quality down gracefully, never block features entirely.

    const getDeviceCapabilities = () => {
  const isMobile = /Android|iPhone/i.test(navigator.userAgent);
  const deviceMemory = navigator.deviceMemory || 4; // not in Safari

  if (isMobile && screen.width < 768)
    return { maxFileSize: 50 * 1024 * 1024, maxDPI: 300, maxPagesPerBatch: 10 };
  if (deviceMemory < 4)
    return { maxFileSize: 100 * 1024 * 1024, maxDPI: 450, maxPagesPerBatch: 30 };
  return   { maxFileSize: 150 * 1024 * 1024, maxDPI: 600, maxPagesPerBatch: 50 };
};

    Memory scales quadratically with render scale — 2× scale = 4× memory. Estimation runs before every heavy operation:

    const estimateMemoryUsage = (pageCount, scale, format) => {
  const basePerPage   = 5 * 1024 * 1024;      // ~5MB at scale 1.0
  const scaleFactor   = Math.pow(scale, 2);    // quadratic growth
  const fmtMultiplier = format === 'png' ? 1.5 : 1.0;
  return pageCount * basePerPage * scaleFactor * fmtMultiplier * 1.5; // 50% safety margin
};

    If the estimate exceeds 50% of available device memory, the user is warned before processing begins — not surprised by a crash mid-operation.

    Three-tier storage architecture

    • RAM — active PDFs and processing buffers. Completely volatile. Close the tab: gone. Zero persistence.
    • IndexedDB — large binary file buffers. localStorage has a 5–10MB limit and stores data as strings, doubling memory cost. IndexedDB stores binary data natively, handles gigabytes.
    • localStorage — metadata only (filenames, timestamps, sizes). Never file content. Used exclusively for the "Recent Files" list.

    Safari download compatibility

    Safari's security model blocks programmatic downloads to prevent drive-by attacks. The HTML5 download attribute works on desktop but fails silently on iOS. Three-method fallback chain:

    const safariCompatibleDownload = (data, filename, mimeType) => {
  // Method 1: download.js library
  if (typeof window.download === 'function') {
    window.download(data, filename, mimeType); return;
  }
  // Method 2: HTML5 anchor element
  try {
    const blob = new Blob([data], { type: mimeType });
    const url  = URL.createObjectURL(blob);
    const a    = Object.assign(document.createElement('a'), {
      href: url, download: filename, style: 'display:none'
    });
    document.body.appendChild(a);
    a.click();
    setTimeout(() => {
      document.body.removeChild(a);
      URL.revokeObjectURL(url); // critical — prevents memory leak
    }, 1000);
    return;
  } catch (e) {}
  // Method 3: window.open() — triggers iOS share sheet
  window.open(URL.createObjectURL(new Blob([data], { type: mimeType })), '_blank');
};

    URL.revokeObjectURL is not optional. Object URLs keep a Blob reference alive until explicitly revoked — for a 50MB PDF, forgetting it means 50MB RAM held for the lifetime of the tab.

    AI Chat — the privacy architecture

    The Chat with PDF feature uses Google's Gemini 2.5 Flash API. The critical design decision: the PDF binary never leaves the device. Only extracted text is transmitted.

    // Step 1: extract text locally with PDF.js
const extractText = async (file) => {
  const pdf = await pdfjsLib.getDocument({ data: await file.arrayBuffer() }).promise;
  let text = '';
  for (let i = 1; i <= pdf.numPages; i++) {
    const content = await (await pdf.getPage(i)).getTextContent();
    text += `\n\n--- Page ${i} ---\n${content.items.map(x => x.str).join(' ')}`;
  }
  return text;
};

// Step 2: stream from Gemini — only text goes to the API
const res = await fetch(
  `https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash:streamGenerateContent?key=${key}&alt=sse`,
  { method: 'POST', body: JSON.stringify({
    contents: [{ parts: [{ text: `Document:\n${extractedText}\n\nQ: ${query}` }] }]
  })}
);

    Why Gemini 2.5 Flash: handles 100-page PDFs (~50,000 tokens) in under 3 seconds, costs ~1/10th of GPT-4, supports streaming. The free tier covers everyday personal use.

    Offline-first with service workers

    After the first page load, every tool works without internet. WASM libraries are cached via service worker — loading in ~20ms vs 500ms+ from CDN.

    self.addEventListener('install', (e) => {
  e.waitUntil(
    caches.open('ihatepdf-v1').then((cache) => cache.addAll([
      '/',
      'https://unpkg.com/[email protected]/dist/pdf-lib.min.js',
      'https://cdnjs.cloudflare.com/ajax/libs/pdf.js/3.11.174/pdf.min.js',
    ]))
  );
});

self.addEventListener('fetch', (e) =>
  e.respondWith(caches.match(e.request).then(c => c || fetch(e.request)))
);

    Test it yourself: DevTools → Application → Service Workers → check "Offline" → reload → every tool still works.

    The engineering philosophy

    • Privacy as architecture, not policy. "We delete your files after 1 hour" requires trusting an organization. "We never receive your files" requires trusting a technical constraint you can verify yourself — open DevTools, watch the Network tab, merge a PDF. Upload column: zero.
    • Device respect. A tool that crashes phones is not a free tool — it's a tool for people with expensive hardware. Adaptive processing ensures every operation works on the device the user actually has.
    • No infrastructure, no limits. Server-based tools limit file sizes and task counts because infrastructure costs money. When processing happens on your device, those constraints disappear. The only limit is your device memory — which is the honest limit.
