How 600 DPI browser
PDF-to-image works

PDF.js, canvas memory management, and device-adaptive processing — the complete engineering story behind high-resolution PDF-to-JPG conversion that runs entirely in your browser tab.

On this page

    Most converters cap at 150 DPI

    Most online PDF to JPG converters cap output at 150 DPI. Some go to 300 DPI if you pay. Very few reach 600 DPI, and those that do require uploading your file to their servers.

    The PDF to JPG converter inside ihatepdf.cv supports up to 600 DPI output — completely free, with zero server upload. Every pixel is rendered locally in your browser. Here's exactly how it works, why 600 DPI matters, and what engineering problems had to be solved to make it work on devices ranging from an iPhone to a 32GB workstation.

    Why DPI matters for PDF-to-image conversion

    DPI stands for dots per inch — it describes how many pixels represent each inch of the original document. The browser's default canvas resolution is 72 DPI: one CSS pixel equals one device pixel at 1× zoom. Everything above that is achieved by scaling the canvas up before rendering.

    DPIUse caseScale multiplier
    72 DPIBrowser base — thumbnail quality1.0×
    150 DPIScreen viewing, social media2.08×
    300 DPIProfessional printing — office standard4.17×
    600 DPIArchival, large-format, engineering drawings8.33×

    The conversion is straightforward — DPI divided by the browser's 72 DPI base produces the required scale multiplier:

    javascript
    const dpiToScale = (dpi) => dpi / 72;

// 150 DPI → 2.08× scale
// 300 DPI → 4.17× scale
// 600 DPI → 8.33× scale

    That 8.33× scale at 600 DPI is where the engineering gets interesting.

    The canvas size problem

    Browsers impose a hard limit on canvas dimensions: 16,384 pixels on most modern browsers (Chrome, Firefox, Safari). At 8.33× scale, a standard A4 PDF page (595 × 842 points at 72 DPI) becomes 4,956 px wide and 7,014 px tall — within the limit. But a legal-size document, wide-format architectural drawing, or landscape slide deck at 600 DPI can easily exceed it.

    The solution is getOptimalScale():

    javascript
    const getOptimalScale = (viewport, requestedScale) => {
  const maxDimension = 16384;
  const testWidth  = viewport.width  * requestedScale;
  const testHeight = viewport.height * requestedScale;

  if (testWidth > maxDimension || testHeight > maxDimension) {
    const scaleFactor = Math.min(
      maxDimension / viewport.width,
      maxDimension / viewport.height
    );
    return scaleFactor * 0.95; // 5% safety margin
  }
  return requestedScale;
};
    Why 0.95?

    The 5% safety margin accounts for browsers that enforce 16,383 rather than 16,384 — an off-by-one that causes silent canvas allocation failures on certain WebKit versions. The cost is negligible; the crash prevention is not.

    Device pixel ratio — the hidden multiplier

    Modern screens have device pixel ratios above 1×. A MacBook Pro Retina display is 2×. Some Android phones are 3×. The render function accounts for this while capping the ratio to prevent memory exhaustion on mobile:

    javascript
    const renderPageToCanvas = async (page, targetScale) => {
  const viewport   = page.getViewport({ scale: targetScale });
  const pixelRatio = Math.min(window.devicePixelRatio || 1, 2); // cap at 2×

  canvas.width  = Math.floor(viewport.width  * pixelRatio);
  canvas.height = Math.floor(viewport.height * pixelRatio);

  const ctx = canvas.getContext('2d', {
    alpha: false,           // white background — JPEG has no alpha channel
    willReadFrequently: false
  });

  ctx.fillStyle = 'white';
  ctx.fillRect(0, 0, canvas.width, canvas.height);
  ctx.scale(pixelRatio, pixelRatio);
  ctx.imageSmoothingEnabled = true;
  ctx.imageSmoothingQuality = 'high';

  await page.render({
    canvasContext: ctx,
    viewport,
    intent: 'print',          // not 'display' — higher quality rendering
    enableWebGL: false,
    renderInteractiveForms: false,
  }).promise;

  return canvas;
};

    Three deliberate decisions in this function are worth unpacking:

    • alpha: false — PDFs have no transparent background. Disabling the alpha channel prevents the browser creating memory it will never use.
    • intent: 'print' — PDF.js has two rendering intents. Print intent uses higher-quality glyph rendering and anti-aliasing, producing noticeably sharper text especially at high DPI.
    • pixelRatio capped at 2× — A 3× ratio on a high-DPI phone would triple memory usage for a visual improvement that's imperceptible at normal viewing sizes.

    Memory management — the real challenge

    This is what separates a tool that actually works from one that crashes your browser tab. At 600 DPI, each A4 page canvas uses approximately:

    Memory math

    4,956 × 7,014 px × 4 bytes (RGBA) ≈ 139 MB of RAM — plus an equivalent amount of GPU texture memory for the canvas, plus PDF.js rendering buffers. For a 50-page document at 600 DPI, the naïve approach allocates ~7 GB. That crashes any browser immediately.

    The solution is explicit canvas disposal after each page. The non-obvious part: simply removing the canvas reference doesn't immediately release GPU memory in most browsers. Setting dimensions to zero forces immediate deallocation.

    javascript
    const canvas = await renderPageToCanvas(page, optimizedScale);

// ... create blob, trigger download ...

canvas.width  = 0;  // ← releases GPU texture memory immediately
canvas.height = 0;
// canvas goes out of scope → GC collects RAM

    Between batches, a deliberate 2-second pause is inserted. Chrome's garbage collector typically triggers after ~1–1.5 seconds of idle time — the pause gives it time to reclaim memory before the next batch begins:

    javascript
    if (batchIndex < batches.length - 1) {
  await new Promise(resolve => setTimeout(resolve, 2000));
  if (window.gc) window.gc(); // hint — browser may ignore
}

    Device-adaptive limits

    The same code runs on a 2GB RAM phone and a 32GB workstation. Rather than applying one-size-fits-all limits, device capabilities are detected and limits adjusted automatically:

    javascript
    const getDeviceCapabilities = () => {
  const isMobile  = /Android|iPhone/i.test(navigator.userAgent);
  const isTablet  = /(tablet|ipad)/i.test(navigator.userAgent);
  const deviceMem = navigator.deviceMemory || 4; // not available in Safari

  if (isMobile && screen.width < 768)
    return { maxFileSize: 50  * 1024 * 1024, maxDPI: 300, maxPagesPerBatch: 10  };
  if (isTablet)
    return { maxFileSize: 75  * 1024 * 1024, maxDPI: 450, maxPagesPerBatch: 25  };
  if (deviceMem < 4)
    return { maxFileSize: 100 * 1024 * 1024, maxDPI: 450, maxPagesPerBatch: 30  };
  return   { maxFileSize: 150 * 1024 * 1024, maxDPI: 600, maxPagesPerBatch: 50  };
};

    A phone user still gets PDF to JPG conversion — just capped at 300 DPI and 10 pages per batch. They get the tool, scaled to what their device can handle.

    Memory also scales quadratically with DPI — doubling DPI quadruples memory because both dimensions double. Before any large conversion, usage is estimated and the user is warned before a silent crash can occur:

    javascript
    const estimateMemoryUsage = (fileSize, pageCount, scale, format) => {
  const baseMemoryPerPage  = 5 * 1024 * 1024;   // 5 MB at scale 1.0
  const scaleFactor        = Math.pow(scale, 2);   // quadratic: 2× scale = 4× memory
  const formatMultiplier   = format === 'png' ? 1.5 : 1.0;
  const estimated          = pageCount * baseMemoryPerPage * scaleFactor * formatMultiplier;
  return { estimated, withSafety: estimated * 1.5 };
};

    JPEG vs PNG — when to use each

    Both output formats are supported. The choice has real consequences:

    • JPEG — lossy compression. Adjustable quality from 60% to 100%. A 300 DPI JPEG at 85% quality is typically 500 KB–2 MB per page. The right choice for photos, scanned documents, and presentations where some quality loss is imperceptible.
    • PNG — lossless compression. No quality setting. A 300 DPI PNG is typically 3–8 MB per page. The right choice for sharp-line documents, text-heavy pages, technical diagrams, or any situation where pixel-perfect reproduction is required. For archival at 600 DPI, PNG is almost always correct.
    PresetDPIJPEG QualityBest for
    Web15085%Social media, email, web embedding
    Print30095%Office printing, CVs, brochures
    Professional50098%High-end printing, detailed documents
    Archival600100%Maximum quality, large format, archiving

    The Archival preset is only shown on devices that can handle it. On phones and tablets, it's hidden entirely.

    Privacy — verifiable, not just claimed

    The entire conversion pipeline runs locally. This is not a policy statement — it's an architectural fact you can verify yourself by opening DevTools → Network tab while converting a file. The upload column stays at zero bytes throughout.

    complete data lifecycle — zero network
    FileReader.readAsArrayBuffer(file) → browser memory
    pdfjsLib.getDocument({ data }) → PDF.js processing (local)
    page.render({ canvasContext }) → Canvas API (local)
    canvas.toBlob('image/jpeg') → Blob in memory
    URL.createObjectURL(blob) → local object URL
    anchor.click() → device storage

    No network requests for file data. Not a claim — a verifiable architectural fact. Open DevTools → Network tab → convert a PDF. Watch the upload column stay at zero.
