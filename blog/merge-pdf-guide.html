pdf-lib, drag-and-drop state management, page-level organization across files, Safari download fallbacks, and the 20-line core that powers it all — zero server dependencies.

On this page

    The moment that started it

    It's late. You have three separate PDF files — a contract, an addendum, and a signature page — and you need them combined before a morning deadline. You search "merge PDF free online", upload your files, and wait. Then you see: "Your files will be deleted after 24 hours."

    You hadn't thought about where your contract was going. You just needed to merge two files. But somewhere between clicking upload and downloading the result, your confidential document traveled to a server you've never heard of, was processed by software you can't inspect, and now sits on a hard drive waiting to be "deleted."

    This is what the ihatepdf.cv merge tool was built to replace.

    Step 1 — file handling without a server

    When you upload to a server-based tool, your browser sends a multipart HTTP request. Those bytes travel over the internet, land on a server, get written to disk, processed, and eventually returned. When you upload to ihatepdf.cv, nothing leaves your browser tab.

    The FileReader API reads the file directly from your local filesystem into an ArrayBuffer — a raw binary representation in memory:

    const readFileAsArrayBuffer = (file) => {
  return new Promise((resolve, reject) => {
    const reader = new FileReader();
    reader.onload  = () => resolve(reader.result);
    reader.onerror = reject;
    reader.readAsArrayBuffer(file);
  });
};

    That ArrayBuffer stays in your browser's memory for the entire session. It is never serialized into a network request. Open DevTools → Network tab → merge a 50MB PDF — the upload column stays at zero.

    Step 2 — the drag-and-drop ordering system

    Drag-and-drop reordering looks simple but involves careful state management. When you drag item 0 toward position 3, removing item 0 shifts everything left — so the target index shifts too. Without adjustment, files land one position too far:

    const handleFileDrop = (e, dropIndex) => {
  e.preventDefault();
  if (draggedIndex === null || draggedIndex === dropIndex) return;

  const newFiles   = [...files];
  const draggedFile = newFiles[draggedIndex];

  newFiles.splice(draggedIndex, 1);

  // If dragging forward, array shortened by 1 — adjust target
  const adjustedDropIndex =
    draggedIndex < dropIndex ? dropIndex - 1 : dropIndex;

  newFiles.splice(adjustedDropIndex, 0, draggedFile);
  setFiles(newFiles);
};

    Step 3 — page-level organization

    The "Organize Pages Before Merge" feature lets you see every page from every PDF, rotate or delete pages, and reorder across files. This requires rendering every page as a thumbnail via PDF.js:

    for (let fileIndex = 0; fileIndex < files.length; fileIndex++) {
  const pdf = await window.pdfjsLib.getDocument({
    data: await readFileAsArrayBuffer(files[fileIndex])
  }).promise;

  for (let pageNum = 1; pageNum <= pdf.numPages; pageNum++) {
    const page     = await pdf.getPage(pageNum);
    const viewport = page.getViewport({ scale: 0.2 }); // small thumbnail
    const canvas   = document.createElement('canvas');
    canvas.height  = viewport.height;
    canvas.width   = viewport.width;

    await page.render({ canvasContext: canvas.getContext('2d'), viewport }).promise;

    allPages.push({
      id: `page-${globalPageId++}`,
      fileIndex,
      pageIndex: pageNum - 1,
      rotation: 0,
      deleted: false,
      thumbnail: canvas.toDataURL(),
    });
  }
}

    Scale 0.2 is the key. Rendering at full scale for thumbnails would consume enormous RAM — a Letter-size page at 72 DPI at 1× scale is already several megabytes. At 0.2×, thumbnails are small enough to store dozens simultaneously.

    Each page carries its fileIndex and pageIndex — pointers back to the original source for the actual merge step.

    Step 4 — the actual merge

    pdf-lib's copyPages copies page objects from source PDFs into the merged document without re-rendering them. This is why merging is fast for large files: it's a structural operation, not a pixel operation.

    const { PDFDocument, degrees } = window.PDFLib;
const mergedPdf = await PDFDocument.create();

// Pre-load all source PDFs once — avoid redundant parsing
const sourcePdfs = await Promise.all(
  files.map(async (file) => PDFDocument.load(await readFileAsArrayBuffer(file)))
);

const activePages = pages.filter(p => !p.deleted);
for (const pageInfo of activePages) {
  const [copiedPage] = await mergedPdf.copyPages(
    sourcePdfs[pageInfo.fileIndex],
    [pageInfo.pageIndex]
  );
  if (pageInfo.rotation !== 0) {
    copiedPage.setRotation(
      degrees(copiedPage.getRotation().angle + pageInfo.rotation)
    );
  }
  mergedPdf.addPage(copiedPage);
}

const pdfBytes = await mergedPdf.save();

    Pre-loading all source PDFs once matters. Without it, you'd reload and parse each source PDF for every page copy — multiplying processing time linearly with page count.

    Step 5 — Safari download compatibility

    Getting bytes from browser memory to the user's filesystem is more complicated than it sounds. Safari's security model blocks programmatic downloads on iOS. Three-method fallback chain:

    const safariCompatibleDownload = (data, filename, mimeType) => {
  // Method 1: download.js (most reliable when available)
  if (typeof window.download === 'function') {
    window.download(data, filename, mimeType); return;
  }
  // Method 2: HTML5 anchor element
  try {
    const blob = new Blob([data], { type: mimeType });
    const url  = URL.createObjectURL(blob);
    const a    = Object.assign(document.createElement('a'), {
      href: url, download: filename, style: 'display:none'
    });
    document.body.appendChild(a);
    a.click();
    setTimeout(() => {
      document.body.removeChild(a);
      URL.revokeObjectURL(url); // prevents memory leak
    }, 1000);
    return;
  } catch (e) {}
  // Method 3: window.open() — triggers iOS share sheet
  window.open(URL.createObjectURL(new Blob([data], { type: mimeType })), '_blank');
};

    The 20-line core

    Strip away all the UI, error handling, and Safari compatibility, and the entire merge operation is this:

    import { PDFDocument } from 'pdf-lib';

async function mergePDFs(files) {
  const merged = await PDFDocument.create();

  for (const file of files) {
    const buffer = await file.arrayBuffer(); // stays in browser memory
    const pdf    = await PDFDocument.load(buffer);
    const pages  = await merged.copyPages(pdf, pdf.getPageIndices());
    pages.forEach(page => merged.addPage(page));
  }

  return merged.save();
  // returns Uint8Array — zero network requests made
}

    Twenty lines. Zero server dependencies. Works in any modern browser. The production version at ihatepdf.cv/merge-pdf adds drag-and-drop ordering, page-level thumbnails, rotation, deletion, Safari compatibility, and memory management — but is architecturally identical to this.
