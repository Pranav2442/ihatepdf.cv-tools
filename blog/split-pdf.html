I Built a PDF Splitter
That Never Touches
Your Files

Privacy shouldn't be a premium feature. How the ihatepdf.cv Split PDF tool works entirely in your browser — no server uploads, no watermarks, no limits. Full engineering breakdown.

On this page

    A familiar 10 PM nightmare

    You have a 40-page report. Your client needs pages 12–18. The only tools you can find either slap a watermark on the output, cap your file size, or — most troublingly — upload your document to some server you've never heard of before "processing" it.

    That last part should bother you more than it does. The split PDF tool on ihatepdf.cv was built to make this problem go away permanently. Not with better marketing language. With a different architecture.

    The server-upload model and why it exists

    Most online PDF utilities share the same uncomfortable architecture: you upload your file, it travels to a remote server, gets processed, and gets sent back. Along the way, your document — which might contain contracts, medical records, or financial data — passes through infrastructure you don't control and likely don't understand.

    Some tools are upfront about this. Many are not. And even the honest ones that promise to "delete your file after 1 hour" are asking you to take their word for it. There is a better way.

    The key insight: Modern browsers are powerful enough to do serious PDF manipulation entirely on the client side — no server required. Thanks to the File API, WebAssembly, and libraries like pdf-lib, every byte of processing can stay on your device.

    How the tool actually works

    When you upload a PDF to the Split tool, here's exactly what happens — open DevTools → Network tab and watch the upload column stay at zero bytes throughout the entire operation.

    1. File is read into memory using the browser's native FileReader API as an ArrayBuffer.
    2. pdf-lib parses the document and counts pages — entirely in your browser tab.
    3. You choose a split method — by page ranges, or one file per page.
    4. New PDF documents are created in memory, pages copied over, byte arrays generated.
    5. Your browser downloads the files directly — no round trip, no server.
    // The entire "upload" step — nothing left your machine const buffer = await readFileAsArrayBuffer(file); const { PDFDocument } = window.PDFLib; const sourcePdf = await PDFDocument.load(buffer); // Splitting into custom ranges const newPdf = await PDFDocument.create(); const copiedPages = await newPdf.copyPages(sourcePdf, pageIndices); copiedPages.forEach((p) => newPdf.addPage(p)); const pdfBytes = await newPdf.save();

    Three lines to "upload." Zero bytes sent over the network. The "server" is your own CPU.

    Two ways to split — designed for real workflows

    The tool offers two modes built to cover the most common real-world scenarios.

    Split by Page Ranges
    ✓Type 1-5, 10-15, 20 — get three separate PDFs, one per range.
    ✓Handles both ranges and single pages in the same input string, comma-separated.
    ✓Perfect for extracting chapters, isolating invoice pages, or sharing a section of a report.
    Individual Pages
    ✓One click — every page becomes its own PDF file automatically.
    ✓A 20-page document becomes 20 separate downloads.
    ✓Downloads are staggered 100ms apart to avoid overwhelming the browser's download manager.

    The range parser handles an important edge case: PDF pages are zero-indexed internally, but humans count from 1. A simple - 1 offset on parsed integers handles this — small detail, meaningful UX impact.

    const parseRanges = (rangeStr) => { const result = []; for (const part of rangeStr.split(",").map((s) => s.trim())) { if (part.includes("-")) { const [start, end] = part.split("-").map((n) => parseInt(n.trim())); if (!isNaN(start) && !isNaN(end) && start <= end) result.push({ start: start - 1, end: end - 1 }); // zero-index offset } else { const num = parseInt(part); if (!isNaN(num)) result.push({ start: num - 1, end: num - 1 }); } } return result; };

    Cross-browser downloads — the Safari problem

    Cross-browser file downloads sound trivial until you try to support Safari on iOS. The tool uses a layered download helper that tries progressively more compatible approaches, falling back gracefully at each step.

    1. A custom window.download function if available in the environment.
    2. The legacy msSaveOrOpenBlob for old Edge and IE.
    3. A programmatically clicked <a download> tag — the standard modern approach.
    4. As a last resort, window.open() with a blob URL, with a prompt to allow pop-ups.

    iOS Safari gets special treatment: a callout banner appears reminding users to check their Files app → Downloads folder, since Safari's native download behavior differs from desktop browsers. The detection is a precise UA check:

    const isIOSSafari = () => { const ua = window.navigator.userAgent; return ( (!!ua.match(/iPad/i) || !!ua.match(/iPhone/i)) && !!ua.match(/WebKit/i) && !ua.match(/CriOS/i) // exclude Chrome on iOS ); };

    Local history — no backend needed

    The tool tracks your recent splits using a hybrid storage strategy that keeps everything local and works offline.

    Storage Architecture
    ✓localStorage — stores lightweight metadata (filename, tool, timestamp, size). Fast reads, instant UI.
    ✓IndexedDB — stores the actual PDF bytes. The browser's structured data store handles large binary blobs without breaking a sweat.
    ⚠History capped at 50 entries — keeps storage usage reasonable. Pragmatic, not precious.
    const addToHistory = async (pdfData) => { const history = JSON.parse(localStorage.getItem(HISTORY_KEY) || "[]"); const newEntry = { id: Date.now(), name: pdfData.name, tool: pdfData.tool, ... }; if (pdfData.bytes) { await dbSet(`pdf_${newEntry.id}`, pdfData.bytes); // IndexedDB } history.unshift(newEntry); localStorage.setItem(HISTORY_KEY, JSON.stringify(history.slice(0, 50))); };

    Zero quality loss — because there's no re-rendering

    One concern people have with PDF tools is quality degradation. This tool sidesteps the issue entirely — because there is no re-rendering happening.

    pdf-lib doesn't rasterize your pages into images and repack them. It copies the raw PDF page objects from the source document into the new one. Text stays as text. Vector graphics stay as vectors. Embedded fonts stay embedded. You get a byte-perfect extraction, not a screenshot of a page.

    No compression. No re-encoding. No quality loss. The split files are structurally identical to the corresponding pages in the original — just in a new document wrapper.

    The privacy guarantee that's structural, not a promise

    When you upload to a server-based PDF tool, here's what happens regardless of their privacy policy:

    1. Your file travels over the internet to their data center. Encrypted in transit — but it exists on network infrastructure between you and them.
    2. The file lands on a server and is written to disk. It lives on hardware you don't control.
    3. At some point the file is "deleted" — assuming deletion systems work correctly and aren't subject to backup retention policies.

    For contracts, medical records, financial statements, or legal filings, this chain of custody has real implications. GDPR, HIPAA, and most professional codes of conduct for legal and financial work have specific requirements about where client data may be processed.

    ihatepdf.cv's answer is not "we delete your files promptly." It is "we never receive your files at all." That is a categorically different privacy guarantee — and it's structural, not a marketing promise. There is no code path that sends your files anywhere.

    The bigger picture

    The Split PDF tool is part of a broader philosophy: your files are yours. The browser has become powerful enough that the reflexive "upload to server → process → download" pipeline is often unnecessary — and increasingly, irresponsible.

    Client-side PDF processing is just one example. The same principle applies to image compression, document conversion, data parsing, and dozens of other utility tasks that millions of people outsource to servers every day when they don't have to.

    As developers, we have a responsibility to ask: does this actually need to leave the user's device? More often than the current ecosystem suggests, the answer is no.
