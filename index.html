How it works

Privacy as architecture,
not policy

Most PDF tools upload your files to servers. ihatepdf.cv doesn't — not because of a privacy policy, but because the architecture makes it technically impossible.

Your file → FileReader API → ArrayBuffer in browser RAM
         → WebAssembly (pdf-lib / Ghostscript / Tesseract)
         → Processed bytes in browser RAM
         → Blob URL → downloads directly to your device

Network requests containing your file data: 0
Verify: DevTools → Network tab → merge a PDF → upload column stays at zero
01
No watermarks, ever
No server generating output means no mechanism to inject a watermark. The output is produced by the WebAssembly library in your browser — it only does what you ask.
02
No task limits
Limits exist to throttle server costs and push users to paid tiers. With no server processing, there is nothing to throttle. Merge 50 PDFs in one afternoon — no friction.
03
Works fully offline
WebAssembly libraries are cached via service worker after the first page load. Disconnect your internet — every tool still works. Impossible for server-based tools.
04
No account required
Accounts exist to track usage against limits. With no limits to enforce and no server-side data to store, there is nothing to log in for.
05
No file size limits
Server tools cap free-tier file sizes because large files cost money to process in the cloud. When processing happens on your device, the only limit is your device's RAM.
06
Professional-grade processing
Powered by Ghostscript, Tesseract, pdf-lib, and PDF.js — the same libraries used in professional desktop software — compiled to WebAssembly and running at near-native speed.
